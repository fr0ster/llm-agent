# Dual-LLM Presentation Stage

**Issue:** [#15 — Performance: pipeline parallelization and dual-LLM tool-loop](https://github.com/AInCoder/llm-agent/issues/15)
**Scope:** Option B — Dual-LLM tool-loop (reasoning + presentation)
**Date:** 2026-03-28

## Problem

The tool-loop uses a single LLM for both reasoning (tool selection) and final response generation. When the LLM presents large tool results (e.g., ABAP source code), it re-generates the content token by token — 25s of generation time for data that already exists in the tool result.

Measured: 38s total request time, 25s spent on `llm_call_3` (presentation).

## Solution

Separate the final response generation into a dedicated `present` pipeline stage that can use a faster/cheaper LLM (e.g., Gemini Flash instead of Claude Sonnet).

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Switch trigger | Always on last iteration (`finishReason !== 'tool_calls'`) | Simple, predictable |
| Fallback | `presentationLlm` not set → `mainLlm` | Backward-compatible, optional feature |
| Prompt | Full assembled context + tool-loop messages | ~15 active tools, minimal overhead; preserves conversation context |
| Presentation system prompt | Configurable via `presentationSystemPrompt` | Flexibility for formatting/tone |
| Builder API | New `.withPresentationLlm(llm)` | Separate from `helperLlm` (different role) |
| Streaming | Presentation LLM streams like mainLlm | No behavior change for consumers |
| Architecture | Separate `present` pipeline stage after `tool-loop` | Maximum transparency for consumers who override pipelines |
| Hardcoded flow | Supported (duplication accepted) | Legacy path must also benefit |

## Changes

### 1. PipelineContext — new fields

```typescript
// Dependencies (injected at creation, read-only)
readonly presentationLlm: ILlm | undefined;

// Mutable state (populated by stages)
/** Content generated by mainLlm in tool-loop's final iteration. */
toolLoopContent: string;
/** Messages accumulated during tool-loop (for presentation LLM context). */
toolLoopMessages: Message[];
```

`toolLoopMessages` carries the full message array from tool-loop (assembled messages + tool call/result pairs) so that the present stage has complete context without rebuilding.

### 2. ToolLoopHandler — conditional yield

When `finishReason !== 'tool_calls'`:

- If `ctx.presentationLlm` is set:
  - Save `content` → `ctx.toolLoopContent`
  - Save `messages` → `ctx.toolLoopMessages`
  - Do NOT yield content chunks (present stage will handle streaming)
  - Yield only the final metadata chunk (usage, timing) without content
- If `ctx.presentationLlm` is NOT set:
  - Yield as before (backward-compatible)

Same logic applies to the hardcoded `_runStreamingToolLoop` in `agent.ts`.

### 3. PresentHandler — new stage handler

```
Reads:  ctx.toolLoopContent, ctx.toolLoopMessages, ctx.presentationLlm, ctx.mainLlm
Writes: yields chunks via ctx.yield()
```

Logic:
1. Select LLM: `ctx.presentationLlm ?? ctx.mainLlm`
2. If `presentationLlm` exists:
   - Build messages: `ctx.toolLoopMessages` + assistant message with `ctx.toolLoopContent` + user message with presentation system prompt
   - Call `llm.streamChat(messages, [], options)` — no tools
   - Yield chunks via `ctx.yield()`
3. If no `presentationLlm`:
   - Yield `ctx.toolLoopContent` as-is (single chunk)
4. Yield final chunk with `finishReason`, `usage`, `timing`

Config (from pipeline YAML):
```yaml
- id: present
  type: present
  config:
    systemPrompt: "Format the response clearly using markdown."
```

`config.systemPrompt` takes precedence over `ctx.config.presentationSystemPrompt`.

### 4. SmartAgentBuilder — new setter

```typescript
private _presentationLlm?: ILlm;

/** Set the presentation LLM for formatting final responses. */
withPresentationLlm(llm: ILlm): this {
  this._presentationLlm = llm;
  return this;
}
```

### 5. SmartAgentConfig — new field

```typescript
/** System prompt override for presentation LLM. */
presentationSystemPrompt?: string;
```

### 6. SmartAgentDeps — new field

```typescript
presentationLlm?: ILlm;
```

### 7. BuilderPromptsConfig — new prompt override

```typescript
/** System prompt for the presentation LLM. */
presentation?: string;
```

### 8. Handler registry — register PresentHandler

```typescript
['present', new PresentHandler()],
```

### 9. Exports

`PresentHandler` exported from `src/index.ts` for consumers who extend or replace it.

## Backward Compatibility

| presentationLlm | present stage | Behavior |
|---|---|---|
| not set | absent | Unchanged — tool-loop yields directly |
| not set | present | present stage yields toolLoopContent without re-generation |
| set | absent (hardcoded) | hardcoded flow runs presentation streamChat after tool-loop |
| set | present | present stage streams through presentationLlm |

## Consumer Usage

```typescript
// Minimal — builder only
const handle = await new SmartAgentBuilder()
  .withMainLlm(claudeSonnet)
  .withPresentationLlm(geminiFlash)
  .withPipeline({
    version: '1',
    stages: [
      { id: 'classify', type: 'classify' },
      { id: 'assemble', type: 'assemble' },
      { id: 'tool-loop', type: 'tool-loop' },
      { id: 'present', type: 'present' },
    ],
  })
  .build();

// With custom presentation prompt
const handle = await new SmartAgentBuilder({
  prompts: {
    presentation: 'Respond in Ukrainian. Use markdown for code blocks.',
  },
})
  .withMainLlm(claudeSonnet)
  .withPresentationLlm(geminiFlash)
  .build();
```

## Expected Impact

- **Estimated saving:** 15-20s on generation-heavy responses (flash models generate 3-5x faster)
- **Cost reduction:** presentation via cheaper model
- **No regression:** fully backward-compatible when `presentationLlm` is not configured
