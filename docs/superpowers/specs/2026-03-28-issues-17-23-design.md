# Design: Issues #17–#23 — Builder, RAG, Retry, SSE, Streaming

**Date:** 2026-03-28
**Issues:** #17, #18, #19, #20, #21, #22, #23
**Status:** Approved

## Overview

Seven production issues reported by a consumer who patches `dist/` at Docker build time via `sed`. All fixes target backward compatibility — no breaking changes to public interfaces.

### Implementation order

1. **#17 + #18 + #19** — Builder + RAG (lowest risk, highest impact)
2. **#22 + #23** — SSE compatibility (simple, isolated)
3. **#20** — RetryLlm decorator (independent)
4. **#21** — Buffer streaming (highest complexity, do last)

## Group 1: Builder + RAG (#17, #18, #19)

### #17: Key-based RAG store lookup

**Files:** `src/smart-agent/builder.ts` (~lines 598, 681)

Replace `Object.values(ragStores)[0]` with explicit key lookup + fallback:

```typescript
const toolStore = ragStores['tools'] ?? Object.values(ragStores)[0];
if (!ragStores['tools']) {
  logger.warn('No "tools" RAG store found, falling back to first available store');
}
```

Apply to both tool vectorization (~line 598) and skill vectorization (~line 681).

**Risk:** If consumer has no `'tools'` key — fallback preserves current behavior. Warning log alerts them.

### #18: Idempotent upsert via metadata.id

**Contract change** in `src/smart-agent/interfaces/rag.ts` — add JSDoc to `IRag.upsert()`:

```typescript
/**
 * If metadata.id is provided, implementations MUST treat it as an
 * idempotent key — repeated upserts with the same id replace the
 * previous record instead of creating duplicates.
 */
```

**`QdrantRag`** (`src/smart-agent/rag/qdrant-rag.ts`, ~line 111):

```typescript
const pointId = metadata?.id
  ? await deterministicUUID(metadata.id)
  : crypto.randomUUID();
```

`deterministicUUID()` — private helper, SHA-256 of id truncated to UUID format.

**`InMemoryRag`** (`src/smart-agent/rag/in-memory-rag.ts`):

```typescript
if (metadata?.id) {
  const existing = this.records.findIndex(r => r.metadata.id === metadata.id);
  if (existing !== -1) this.records[existing] = newRecord;
  else this.records.push(newRecord);
} else {
  this.records.push(newRecord);
}
```

**`VectorRag`** — analogous to InMemoryRag (overwrite by metadata.id match).

**Risk:** Existing Qdrant points with random UUIDs won't be cleaned up automatically. Consumers with persistent stores need a one-time cleanup or TTL. Custom `IRag` implementations (HANA, Postgres, etc.) should follow the updated contract.

### #19: Per-item error handling in vectorization loop

**File:** `src/smart-agent/builder.ts` (~lines 598–609, 681–692)

`IRag.upsert()` already returns `Result<void, RagError>` — check `.ok` instead of letting exceptions propagate:

```typescript
for (const t of toolsResult.value) {
  const result = await toolStore.upsert(...);
  if (!result.ok) {
    logger.warn(`Tool vectorization failed for "${t.name}": ${result.error.message}`);
  }
}
```

Same pattern for skill vectorization loop.

**Risk:** Low. One failed tool no longer kills the entire MCP connection setup.

## Group 2: LlmAdapter Retry (#20)

### New decorator: `RetryLlm`

**File:** `src/smart-agent/resilience/retry-llm.ts`

Implements `ILlm`, wraps inner `ILlm`. Follows existing decorator pattern (`CircuitBreakerLlm`, `FallbackRag`).

```typescript
export interface RetryOptions {
  maxAttempts: number;   // default: 3
  backoffMs: number;     // default: 2000
  retryOn: number[];     // default: [429, 500, 502, 503]
}
```

**`chat()`** — full retry with exponential backoff:

```typescript
for (let attempt = 0; attempt <= maxAttempts; attempt++) {
  if (signal?.aborted) return abortedResult;
  const result = await inner.chat(messages, tools, options);
  if (result.ok || attempt === maxAttempts || !isRetryable(result.error)) return result;
  await backoff(attempt, signal);
}
```

**`streamChat()`** — retry only if 0 chunks yielded:

```typescript
for (let attempt = 0; attempt <= maxAttempts; attempt++) {
  let chunksYielded = 0;
  for await (const chunk of inner.streamChat(...)) {
    if (chunk.ok) { chunksYielded++; yield chunk; }
    else {
      if (chunksYielded === 0 && isRetryable(chunk.error) && attempt < maxAttempts) break;
      yield chunk; return; // already yielded data, can't retry
    }
  }
  if (chunksYielded > 0) return;
  await backoff(attempt, signal);
}
```

**`isRetryable()`** — checks if `LlmError.message` contains any of the configured status codes.

### Composition order

```
RetryLlm → CircuitBreakerLlm → LlmAdapter
```

Retry outside circuit breaker — so retry attempts don't count as separate failures.

### Configuration

Add to `SmartAgentConfig`:

```typescript
retry?: {
  maxAttempts?: number;
  backoffMs?: number;
  retryOn?: number[];
};
```

Builder creates `RetryLlm` wrapper when `retry` config is present.

**Risks:**
- `streamChat()` retry only when 0 chunks yielded — prevents content duplication
- AbortSignal checked before each attempt
- Interaction with CircuitBreaker: retry is outer layer, CB doesn't see retries as separate failures

## Group 3: SSE Compatibility (#22, #23)

### #22: Error in chat.completion.chunk format

**Files:** `src/smart-agent/smart-server.ts` (~lines 891–895), `src/smart-agent/server.ts` (analogous)

```typescript
// Was:
res.write(`data: ${jsonError(chunk.error.message, 'server_error')}\n\n`);

// Becomes:
res.write(`data: ${JSON.stringify({
  id, object: 'chat.completion.chunk', created, model: responseModel,
  choices: [{
    index: 0,
    delta: { content: `[Error] ${chunk.error.message}` },
    finish_reason: 'stop',
  }],
})}\n\n`);
```

**Finish reason safety net:** After the stream loop, if no `finish_reason` was sent:

```typescript
if (!finishReasonSent) {
  res.write(`data: ${JSON.stringify({
    ...baseResponse,
    choices: [{ index: 0, delta: {}, finish_reason: 'stop' }],
  })}\n\n`);
}
```

**Risk:** Clients parsing raw `{"error":...}` will break — but issue confirms Cline/Goose/Continue expect chunk format. Apply to both `smart-server.ts` and `server.ts`.

### #23: Suppress usage stats

**Config addition** in `SmartServerConfig`:

```typescript
reportUsage?: boolean; // default: true
```

**Change in `smart-server.ts`:**

```typescript
// Was:
if (body.stream_options?.include_usage && lastUsage) {

// Becomes:
if (this.config.reportUsage !== false
    && body.stream_options?.include_usage && lastUsage) {
```

**Risk:** Low. Default `true` preserves current behavior. Internal telemetry/spans unaffected.

## Group 4: Buffer Streaming (#21)

### Config option `streamMode`

Add to `SmartAgentConfig`:

```typescript
streamMode?: 'full' | 'final'; // default: 'full'
```

- `'full'` — current behavior, all chunks streamed immediately
- `'final'` — intermediate tool loop iterations buffered, only final response streamed

### Change in `_runStreamingToolLoop()`

In `'final'` mode, buffer content per iteration. Yield only when `finishReason === 'stop'` (final iteration):

```typescript
if (this.config.streamMode === 'final') {
  iterationBuffer += chunk.content;
  // After iteration ends:
  if (finishReason === 'stop') {
    yield { ok: true, value: { content: iterationBuffer } };
  } else {
    iterationBuffer = ''; // tool calls follow, discard
  }
}
```

### Chunk behavior in `'final'` mode

| Chunk type | Behavior |
|-----------|----------|
| Content (intermediate iteration) | Buffered, discarded |
| Content (final iteration) | Streamed normally |
| External tool calls | Streamed always (consumer executes them) |
| Heartbeat | Streamed always (prevents timeout) |
| Presentation LLM output | Streamed normally (already final) |
| Usage / finish_reason | No change |

### Buffer limits

No separate limit needed. Buffer holds one iteration's content, reset before next iteration. Max size bounded by provider's `max_tokens`.

**Risks:**
- Client sees no content during tool loop (30–60s). Heartbeat comments already exist and prevent timeout.
- Presentation LLM: in `'final'` mode, only presentation output streams — intermediate content is discarded. Needs careful testing.
- External tool calls stream always — consumer must handle them even when content is buffered.
